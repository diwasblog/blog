Types of Contraints

NOT NULL Field must have a value
UNIQUE Value must not already be in table 
PRIMARY KEY Must have value and be unique , used to identify record in a table
FOREIGN KEY All values must exist in another table.
CHECK Checks that all values meet condition
DEFAULT If no value provided, value is set to the default. 
  
    //It is recommended that you use lower case names for the table and as well as all the columns.


    CREATE TABLE IF NOT EXISTS employee ( 
      id SERIAL PRIMARY KEY, //primary constraint     -- serial data type is auto incrementing four-byte integer
      // id SERIAL BIGSERIAL 
      name varchar(40) NOT NULL,
      pass CHARACTER VARYING,
      first_name CHARACTER varying(30),
      COUNTRY_NAME varchar(40) NOT NULL,
      username text,
      age numeric ,
      sex boolean Default FALSE,    // Oracle = boolean , sql server = bit
      region_id decimal(10,0) NOT NULL,
      SALARY decimal(8,2) DEFAULT NULL, 
      mail character varying(255) NOT NULL default 'xyz@gmail.com'::character varying,
      JOB_TITLE varchar(35) NOT NULL DEFAULT ' ',
      min_salary decimal(6,0) DEFAULT 8000,  // default contraint
      active boolean NOT NULL DEFAULT true,
      country_id integer,
      start_date TIMESTAMP WITHOUT TIME ZONE,   //without timezone acts like UTC time   
      date_create timestamp with time zone DEFAULT now()
      department_id integer,
      created_on timestamp without time zone,
      foreign key(country_id) REFERENCES  country(id),         //foreign key constraint
      foreign key(department_id) REFERENCES  department(id),  //foreign key constraint

      my_ip_vfour_address INET,                                 // 10.0.2.8/16
      my_mac_address MACADDR,                                   //08:00:2b:01:02:03

      cost integer default 50 constraint practices_cost CHECK (cost >= 0 AND cost <= 1000) //check and default constraint
      UNIQUE(name,sex)
      );

For generating UUID

uuid_generate_v1() method call can be made automatically to generate a default value for any newly inserted row. When defining the column.This function generates a version 1 UUID. This involves the MAC address of the computer and a time stamp. Note that UUIDs of this kind reveal the identity of the computer that created the identifier and the time at which it did so, which might make it unsuitable for certain security-sensitive applications.
uuid uses 16 bytes per value, which is twice what is used by int8.

Step 1 : Create uuid_generate_v1() function

          
              -- FUNCTION: public.uuid_generate_v4()
      
              -- DROP FUNCTION public.uuid_generate_v4();
              
              CREATE OR REPLACE FUNCTION public.uuid_generate_v4(
                )
                  RETURNS uuid
                  LANGUAGE 'c'
              
                  COST 1
                  VOLATILE STRICT 
              AS '$libdir/uuid-ossp', 'uuid_generate_v4'
              ;
              
              ALTER FUNCTION public.uuid_generate_v4()
                  OWNER TO postgres;

Step 2 : Then create table with uuid column

            
              
              CREATE TABLE IF NOT EXISTS employee (

                pkey UUID NOT NULL DEFAULT uuid_generate_v1(), 
                CONSTRAINT pet_pkey_ PRIMARY KEY (pkey)
                );

    SET TIMEZONE TO 'Japan';
    SELECT '2011-01-01 00:00:00'::TIMESTAMP;    //  2011-01-01 00:00:00
    SELECT '2011-01-01 00:00:00'::TIMESTAMP WITH TIME ZONE;   // 2011-01-01 00:00:00+09     where 9 is UTC OFFSER timezone , for Nepal : 05
    SELECT '2011-01-01 00:00:00+03'::TIMESTAMP;               //   2011-01-01 00:00:00

  

  
    TIMESTAMP WITHOUT TIME ZONE   =====> NO UTC TIME
    TIMESTAMP WITH TIME ZONE    =======> WITH UTC TIME

Why UUID is good as compared to auto-increment Id?

  • At scale, when you have multiple databases containing a segment (shard) of your data, for example, mobile and web save data to same table then using a UUID mean that one ID is unique across all databases.
  • Also it is 16 bytes data which is globally unique across all the databases, this makes moving of data safer.
  • UUIDs do not reveal information about your data, so it would be safer to use in a URL, for example. If I am customer 12345678, it’s easy to guess that there are customers 12345677 and 1234569, and this makes for an attack vector.

Drawback of UUID

  • Its size is larger(16 bytes) then general auto-increment field like int,bigint, etc. Its sizer is largest in SQL server

 

So use GUID only you need it.
