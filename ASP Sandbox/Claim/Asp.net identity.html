Type (Mode) of Authentication

  • Forms (Via username and password provided by developer if correct then authenticated)
  • Windows (authenticate via windows OS username and password)
  • Passport (authenticate via microsoft passport )
  • Federated
  • None

Ways to implement Authentication

  1. Form Authentication
  2. Asp.Net Identity

Form Authentication is common in 2005 and after that Asp.Net Identity is started to use because at that time social media is raising its level. So that asp.net identity provide facility to login not only via form but also via social media login. Asp.Net Identity is just advance form of Form Authentication.

Asp.Net Identity

  • Identity is a .Net Web based framework
  • Identity provides security to website.
  • Can used in web form ,mvc,web api ,signalR
  • It has own database ,own code
  • Security such as : Login ,Signup,Forget Password, Change Password,Login with social Media(thirdparty),Send Token,Encrypt Password
============================================================================================================

Advantage/Goal of Asp.Net Identity

  • One Asp.Net Identity System( One Identity System but can be used in all like webform,mvc,signalR,web api)
  • Ease of plugging in profile data about the user.
  • Persistence Control : Asp.net Identity has its own database for keeping all user information at persistance level(for long term). It uses EF code first approach to implement all of its persistence mechanism.It can be used with NoSQL,Azure Storage Table Service,Sharepoint and many more.
  • Role Provider: You can restrict access to parts of your application by roles. We can easily create roles such as "Admin" and add users to roles.
  • Claims Based:Asp.net Identity supports claims-based authentication
  • Social Login Provider:Can easily add social log-in .
  • OWIN Integration:ASP.NET Identity uses OWIN Authentication for log-in/log-out of users in the web site. This means that instead of using FormsAuthentication to generate the cookie, the application uses OWIN CookieAuthentication to do that.Owin is now part of our application, and Identity 2.0 relies upon it to deliver the goods. So, owin acts as middleware.

    Previously , for cookie based authentication we used Form Authentication and for external identity providers we used DotNetOpenAuth but this is replaced by OWIN authentication middleware.Good this of using OWIN is that we should not relay on IIS as web host.It provides its own cookie authentication (same as Form Authentication) where user is validate on the basis of cookie on each and every request send to web server. One things OWIN cookie have that Form based cookie doesnot have is Claims.

    Also OWIN Authentication has external authentication that Form authentication doesnot have.
