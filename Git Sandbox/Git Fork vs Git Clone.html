Git Terminology

 

 

 

Working Area

  • The Working Area is the area where you are currently working. It is where your files live. This area is also known as the “untracked” area of git.
  • Any things in working Area if not saved in git will not be saved to git.

Check a file in Working Area: This is unstaging area ie. git isnot tracking this file.

Staging Area:

  • The Staging Area is when git starts tracking and saving changes that occur in files.
  • These saved changes reflect in the .git directory
  • After you type: git add .  will goto staging area.

Local repository

  • The git command git commit takes all changes in the Staging Area, wraps them together and puts them in your Local Repository. 
  • After commit , stash area will be empty.

Remote Repository

This is remote located repository.

 

Git Fork vs Git Clone

 

Central Repository : Say Company Repo. Called as UpStream Repo

Personal Repository: When you fork a repository then , it will get copied to Personal GitHub. Called as Origin Repo

Local Repository : When you clone any remote repository then will get copied to Local Work Space.

    
      git push origin master  // This tells pushing the code to Origin Repository.
Fork Clone
Forking is done on the GitHub Account  Cloning is done using Git. 
When you fork a repository then , it will get copied to Personal GitHub.ie .Remotely located When you clone any remote repository then will get copied to Local Work Space.

 

Use of Pull Request ( Used if you want to be contributor in Central Github)

Changes made to the forked repository can be merged with the original repository via a pull request. Pull request knocks the repository owner and tell that “I have made some changes, please merge these changes to your repository if you like it”. On the other hand, changes made on the local machine (cloned repository) can be pushed to the upstream repository directly. For this, the user must have the write access for the repository otherwise this is not possible. If the user does not have the write access, the only way to go is through the forked request. So in that case, the changes made in the cloned repository are first pushed to the forked repository and then a pull request is created. It is a better option to fork before clone if the user is not declared as a contributor and it is a third party repository (not of the organization).

Suppose you want to be vue contributer then just fork their repository and this will bring to my Remote Repository .From my Remote Repository clone to my Local Repostiory. Then make changes in local repository and push it to my Own Remote Repository .Then if you want to contribute to vue project, then just create "Pull Request" request to Central Repository ie. it is saying “I have made some changes, please merge these changes to your repository if you like it”. Then vue head person just review the pull request and if it is useful then
