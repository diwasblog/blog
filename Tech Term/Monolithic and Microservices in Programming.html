What is Modular Application Development?

In modular Application Development, we create multiple independent modules of software and later combine all to make a large application. Here each module is independent and can be reused in other applications as well. Finally, all the modules are smushed/combined in one application and deployed in one machine. Eg. Installed Photoshop, each has a separate copy on their system.

Then came web application, here follow the same modular techniques but hosted on a web server so that multiple users use the same application from the server. Eg. Web Photoshop

These both desktop and web applications follow monolithic architecture. Ie .Single Store Architecture. 

Biggest Disadvantage of Monolithic Architecture is

1) " Bigger the application, difficult to deploy it."

Suppose you have made an application and if say in module 4 small changes has to be made then we have to build whole system and have to check whether  module4 has effect on other module or not. And if no negative effect, then publish the application.

2) Scalability

Say e-commerce sites, we know that eCommerce sites has large spikes on a certain day.

If you have eCommerce sites then if traffic has raised on your sites then you will scale the whole application server has to scale. 

Suppose say if only in certain modules has much traffic then also we have to scale whole application. So, lots of money may be required to duplicate the instances of whole application.

3) Single Point of Failure

If one modules failed then will not work another module as well because we have to publish on single system

Microservices Application

Then came microservices architecture, here we have self-working modules(mini-application) and each of these modules is hosted on a separate server. So, if we have more traffic on particular module then we can just scale its server. And shouldnot have to touch other servers. So, it minimizes the cost.

 

Here each mini-application can communicate with each other via APIs and together works as a bigger application.

 

So this mini-application (self-completed application) which are hosted on different servers and can communicate with each other via APIs.So this mini-application are called as microservices.

Advantages of Microservices:

1) Deployment Flexibility

Different developers can work on each mini-application and can be flexible to store in different servers. These mini-application can communicate via APIs(Language Independent) and makes a complete system.

2)Language Flexibility

This mini-application can be built in any language and these applications can communicate with Apis easily. And finally forms a big application.

3)Can Be Scaled Separately

Each mini application can be scaled separately

Disadvantages of Microservices:

1) Deployment/Architecture Complexity

Can to make an architecture to host in separate server and communicate each other efficiently.

Microservices are smaller single responsibility services. Which does only one thing and one thing only.And have clear boundary.Usually they have their own data and datastore.

Logic, as well as data of a single responsibility microservice, should not leaks outside of it. 

Here we have divided it into 4 services (product, inventory, user, order). These are separate projects.And when inventory needs to communicate with the product then it will communicate via API.

Each of these services can be hosted in different web servers as well as database servers.

What about cost of Deployment ?

COST of IIS 

  • If all the services are deployed in the same windows machine in an IIS, if IIS has an issue then all apps will fail
  • If you want multiple IIS and boxes,then there will be a significantly high cost.

COST of Database

  • The database should be isolated from each other.
  • SQL Server may not be cost-effective because it has to license cost.

Solution: Container is the solution for all. The container is used for packaging software and all its dependencies into a standard unit of development, deployment, and shipment. They are immutable ie. Container images of one service can be copied into multiple machines in no time and cost without creating a problem for others.

You can do the same with a virtual machine

When should I use Microservices?

  • Start with the Monolithics application if you are not sure.
  • The main decider of going into microservices is your agility of deployment. Think about which functionality makes sense breaking up for ease of deployment.
  • If you are going with Microservices, docker is your best friend.

Tools Needed

For starting microservices, we do not need any tools but try using Docker 

  • Docker
  • When you use docker, there comes the question of how to scale out or scale down. Hence we need an Orchestration Engine. Kubernetes is most popular open source container orchestration engine.

Microservice Communication (Between services and services)

  • Most Popular is REST API (gRPC is growing nowadays).
  • Message Based Communications(Queue or Pub/Sub).Which works very well if you want to offload the work to some background microservices.And the response is not time sensitive.Eg. Banking Statement for last 1 year.

What about Reports ?

When data is available across multiple databases then getting a consolidated view of data or report is difficult.

So, what you can do is :

  • You can either use data aggregation by calling multiple services.
  • Or you can create a data stream for all the data and have an aggregated create read-only view of the data.
  • Use API Gateway (octelet ) which is the best one.

Few Notes for building microservices application

  • When you start building microservices, think about resilience during design.
  • Establish a communication pattern you have to use between your microservices-based on response needs.
  • Logging and Monitoring is a critical aspect of microservices. Cloud-Based Centralizing logging is extremely critical. And passing correlation log id across microservices will save you a lot of debugging time in the future.
